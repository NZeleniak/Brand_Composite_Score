Brand Composite Score Documentation

This page explains how the standalone report works, how ReviewTrackers data is refreshed, which local scripts are used, and what environment values are required.

Methodology

The refresh script injects ReviewTrackers data into the report; the browser recalculates the views locally.

Calculation

rating_100 = (rating_raw / rating_scale) * 100

volume_weight = ln(1 + review_count)

weighted_score =
sum(rating_100 * volume_weight * recency_weight * source_weight)
/
sum(volume_weight * recency_weight * source_weight)

brand_composite =
(resident_experience * 0.70)
+ (employer_brand * 0.20)
+ (nps_component * 0.10)

In V1, Resident NPS replaces the missing Trust/Friction source for the 10% component. Employer Brand combines Employee NPS with Glassdoor and Indeed employer reviews.

Governance Rules

  • Public review ratings are normalized to a common 0-100 scale.
  • Volume weighting uses diminishing returns so large residences do not dominate unfairly.
  • Recency weighting keeps the score current without discarding older signals.
  • Confidence grades use review volume and distinct-source coverage.
  • Google-only scoring is retained as a bias and directional-accuracy control.

How It Works

The project stays static while the local scripts do the data refresh work.

Report Page

  • Brand Composite Score.html is the dashboard page.
  • It contains embedded CSS, JavaScript, and the latest generated data.
  • It can be opened directly from the filesystem.

Refresh Script

  • Reads ReviewTrackers API data using a local token.
  • Aggregates reviews by residence and source.
  • Captures Glassdoor and Indeed as employer review snapshots, while excluding them from residence review scoring.
  • Finds the newest Property_Data_Sheet_YYYY.csv and joins it by residence name.
  • Embeds resident NPS, employee NPS, occupancy, and mapping diagnostics for the optional residence-data mode.
  • Injects the normalized data into the report file.

No Database

  • No Snowflake or other database is required.
  • No API credential is placed in the browser page.
  • The report recalculates filtered views locally.

Data Modes

The report opens in Reviews mode. Survey data is included only when a user selects Survey or Reputation score.

Mode Buttons

  • Reviews uses ReviewTrackers metrics and performance score only.
  • Survey uses resident NPS, employee NPS, and occupancy only.
  • Reputation score uses the Word-document model: Resident Experience 70%, Employer Brand 20%, and Resident NPS 10%.
  • Reputation score also displays occupancy in the table for context, but occupancy is not included in the score calculation.
  • Reputation score table rows can be sorted by Reputation Score or Confidence.
  • Combined score remains in the code but is hidden from the dashboard for now.
  • The filter controls show Platforms, Region, Locations, Sources, and date range.
  • Platforms are backed by ReviewTrackers groups. Region is a visible placeholder only in this version.

Survey And Reputation Formulas

NPS values are normalized from -100..100 to 0..100 before averaging.

resident_nps_score = (resident_nps + 100) / 2
employee_nps_score = (employee_nps + 100) / 2
survey_score = average(resident_nps_score,
                       employee_nps_score)
total_score = average(reviewtrackers_score,
                      resident_nps_score,
                      employee_nps_score)
employer_review_score = Glassdoor + Indeed weighted score
employer_brand = average(employee_nps_score,
                         employer_review_score)
nps_component = resident_nps_score
document_logic_score =
  resident_experience * 0.70
+ employer_brand * 0.20
+ nps_component * 0.10

Mapping Diagnostics

  • Property sheet rows are matched to ReviewTrackers residences by normalized residence name only.
  • In Survey or Reputation score mode, the report shows matched and unmatched counts.
  • Unmatched ReviewTrackers residences are listed so names can be corrected in the source data or future override file.

Occupancy Correlation

  • The correlation value is Pearson correlation between occupancy average and the active mode score.
  • No chart is used in this version; the report shows a KPI and mapping diagnostics.
  • Survey mode requires occupancy, resident NPS, and employee NPS. Reputation score also requires residence review data and NPS.

API Connection

All ReviewTrackers calls are read-only except creating the temporary authentication token.

Authentication

The local scripts authenticate with ReviewTrackers using a temporary 7-day token tied to this User-Agent:

ChartwellBrandCompositeScore/1.0

The token is saved locally as reviewtrackers-token.json and is ignored by Git.

Read Endpoints

  • GET /v2/reviews
  • GET /locations
  • GET /groups
  • GET /items
  • GET /location_score/{account_id}/locations
  • GET /metrics/overview/breakdown

The default all-filter dashboard cards prefer these ReviewTrackers aggregate endpoints. Filtered residence, group, and source views use the generated local data when an exact native aggregate is not available.

Environment Setup

Create this file locally only. Do not commit credentials or tokens.

.env

REVIEWTRACKERS_EMAIL=your.email@chartwell.com
REVIEWTRACKERS_PASSWORD=your_password

# Optional:
REVIEWTRACKERS_ACCOUNT_ID=account_id
REVIEWTRACKERS_TOKEN=existing_token

Git Safety

The project ignores these local secret files:

.env
reviewtrackers-token.json

Before pushing to GitHub, confirm they are not staged.

Scripts To Run

Use these commands from the project folder.

Create Or Renew Token

node scripts\reviewtrackers-auth.mjs create-token

Refresh The Report

node scripts\reviewtrackers-refresh-report.mjs --published-after 2024-05-21 --published-before 2026-05-21

After the command finishes, refresh Brand Composite Score.html in the browser.

Property Data Sheet

This file is manually replaced each year and joined into the report during refresh.

File Naming

Use this naming pattern:

Property_Data_Sheet_2026.csv
Property_Data_Sheet_2027.csv

The year can change. The script automatically uses the newest matching file it finds.

Replacement Flow

  • Remove or replace the old property data sheet.
  • Add the new file using the same naming pattern.
  • Run the refresh script.
  • Refresh the report page in the browser.
  • Use Survey or Reputation score to include the sheet in scoring and correlation analysis.

Files And Limits

These are the important project files and known gaps.

Generated And Support Files

  • Brand Composite Score.html: report page.
  • Documentation.html: documentation page.
  • data/reviewtrackers-report-data.json: normalized generated data.
  • Property_Data_Sheet_YYYY.csv: manually replaced annual property sheet.
  • data/residence-master.json: optional metadata override.
  • scripts/reviewtrackers-auth.mjs: token helper.
  • scripts/reviewtrackers-refresh-report.mjs: API refresh and report injection.

Current Limitations

  • Employer Brand is used only in Reputation score and comes from Employee NPS plus Glassdoor/Indeed when those reviews are present.
  • BBB/trust/friction remains out of scope until a real source is added; Reputation score uses Resident NPS for that V1 10% component.
  • Outside operating-region groups means the residence has ReviewTrackers groups, but not one of the operating-region groups used for validation.
  • Competitor endpoints are optional and only needed for future benchmarking.
  • The scripts do not update reviews, locations, responses, or statuses.
